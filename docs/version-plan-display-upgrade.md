# Agent Light 版本计划：屏幕升级

## 当前状态

- 状态：等待硬件到货
- 目标版本：`vNext`
- 创建日期：2026-07-07
- 范围：增加屏幕辅助状态展示，同时保持当前 LED 方案兼容

## 背景

当前固件和命令链路只围绕三种灯光状态构建：

- `idle`
- `thinking`
- `running`

这足以提供粗粒度的状态感知，但不足以准确区分 Codex 和 Claude Code hooks 暴露出来的更丰富的 agent 生命周期状态。

## 决策

现阶段不立即替换当前仅使用 LED 的设计。

在屏幕硬件到货后，采用分层状态输出方案：

- 屏幕作为主要状态展示界面
- LED 作为快速余光感知指示
- 蜂鸣器作为少量关键事件的低频提醒通道

## 采用这一方向的原因

- 屏幕可以直接显示精确语义状态，例如 `awaiting_permission`、`tool_running`、`subagent_running`、`task_completed`。
- LED 依然适合远距离和不需要阅读文字时的快速感知。
- 蜂鸣器只适合少数需要立刻注意的事件，不适合用于每一次状态切换。

## 建议的硬件方向

首选第一版实现方案：

- `128x64` I2C OLED 显示屏
- 保留现有两组三色 LED
- 增加一个可控蜂鸣器，并支持静音

初步 ESP32 引脚规划：

- 显示屏：优先使用 `GPIO21`（`SDA`）和 `GPIO22`（`SCL`）作为 I2C
- 蜂鸣器：优先使用空闲通用输出脚，例如 `GPIO32` 或 `GPIO33`

最终引脚选择必须在实际屏幕模块和蜂鸣器硬件到货后重新确认。

## 产品目标

- 保持现有 hook 与桥接服务工作流不变。
- 在不破坏当前 LED 行为的前提下，提高可见状态的精细度。
- 同时支持 Codex 和 Claude Code 的更丰富事件映射。
- 让第一版带屏幕的发布保持足够简单，便于在 ESP32 上调试。

## 本阶段非目标

- 硬件到货前不立即重写固件。
- 第一版升级不引入彩色 LCD 或触摸 UI。
- 不为所有状态切换都增加持续的声音提示。
- 除非必要，不修改现有主机桥接协议到破坏兼容的程度。

## 计划中的状态模型

硬件到货后的目标状态方向：

- `idle`
- `thinking`
- `tool_running`
- `awaiting_permission`
- `tool_failed`
- `subagent_running`
- `compacting`
- `task_completed`
- `stopping`

最终保留哪些状态，应根据稳定且值得展示的 hook 事件来确定。

## 输出层映射

屏幕：

- 精确的 agent 名称
- 精确的状态名称
- 最近一次事件
- 可选的传输链路 / 桥接服务摘要

LED：

- 将详细状态压缩成少量类别
- 建议类别：空闲、活跃、等待人工、错误

蜂鸣器：

- 仅在 `awaiting_permission` 时提醒
- 仅在任务完成时提醒
- 仅在失败或异常停止时提醒

## 实施阶段

### 阶段 0：规划

- 记录升级路线
- 保持当前仅 LED 的实现不变
- 等待屏幕硬件到货

### 阶段 1：硬件拉起

- 确认实际屏幕控制器型号和库选型
- 验证供电、电压和接线方式
- 确认最终 GPIO 分配
- 测试屏幕刷新和蜂鸣器驱动行为

### 阶段 2：固件扩展

- 增加显示驱动抽象层
- 增加蜂鸣器驱动抽象层
- 将“语义状态”与“物理输出行为”解耦
- 保持现有 LED 命令向后兼容

### 阶段 3：主机侧映射升级

- 将 Codex hook 映射扩展到三态模型之外
- 将 Claude Code hook 映射纳入同一套语义状态层
- 定义一张共享的“事件 -> 状态”映射表

### 阶段 4：验证

- 验证现有 LED 行为仍可正常工作
- 验证 hook 高频切换时屏幕内容是否稳定
- 验证蜂鸣器只在选定事件上触发
- 验证 USB、蓝牙、Wi-Fi TCP 三种传输方式下的行为

## 待确认问题

- 到货的实际显示模块是什么：OLED、TFT 还是其他控制器？
- 蜂鸣器是否需要 PWM 音调模式，还是只需要短促提示音？
- 屏幕是否需要同时显示 Claude 和 Codex，还是只显示最近活跃的 agent？
- 是否需要状态历史，还是只显示当前状态？

## 硬件适配的进入条件

仅在以下条件满足后启动实现：

- 实际显示模块已经到手
- 已确认其控制器型号
- 已确认供电与接口模式
- 已确认蜂鸣器的电气特性

## 对当前仓库的影响

在硬件到货前，这份计划不要求修改以下内容：

- 固件命令协议
- 主机 hook 客户端
- Windows 桥接服务
- 当前 LED 控制逻辑

## 下一步动作

屏幕硬件到货后，从阶段 1 开始，并将本计划转换成具体的实现任务清单。
